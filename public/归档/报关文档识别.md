# 报关文档识别

## 任务目标

利用视觉大语言模型，对海关报关文档进行字段识别，以实现自动化填报

## 任务内容

以下以单个客户的文档识别为例

### 文档提取

对于"OCR识别报文.json"文件，需要从content.operateImage中提取多个图片（imageUrl）及其对应编号（imageId）和文件类别（attTypeCode）

**实现**: [`core/data_loader.py`](customs_ocr/core/data_loader.py:1) 中的 [`load_input_data()`](customs_ocr/core/data_loader.py:11) 函数

### 调用视觉大模型API进行识别

对于单张图片，根据其文件类别（attTypeCode），套入不同的prompt模板（包含了不同的字段标准名称池），以指定需要识别的字段，使模型严格输出下列JSON格式：
```json
{
  "preDecHead": [
    {
      "keyDesc": "xx",
      "value": "yy",
      "pixel": [a, b, c, d]
    }
  ],
  "preDecList": [
    [
      {
        "keyDesc": "xx",
        "value": "yy",
        "pixel": [a, b, c, d]
      }
    ]
  ]
}
```
1. preDecHead表示文件的表头，同一keyDesc只能出现一次；preDecList表示文件的表体，可能囊括多个商品，因此同一keyDesc可能存在多次
2. keyDesc表示识别字段的标准名称，必须在prompt的字段标准名称池中，但并不是所有字段都会出现。对于图片中未识别到的字段，不应出现在JSON中。
3. value表示识别字段的对应内容
4. pixel表示识别内容的矩形框相对坐标值，介于[0, 999]之间
5. 图片中的相应字段不一定与标准名称一致，例如对于"运抵国"这一标准名称，它在图片中的识别依据可能是"TO USA"，此时应按照{"keyDesc": "运抵国", "value": "USA", "pixel": "[a, b, c, d]}的形式填写
6. "报关单识别资料分类及字段提取说明_新.xlsx"文件的"资料分类"sheet给出了attTypeCode的对应文件名，"字段识别来源"sheet则给出了不同文件的所囊括的字段标准名称池（keyDesc均为中文，括号中的英文如mainBillNo为key，在后续会用到，因此需要保存keyDesc与key的对应关系）
7. 参考下面的内容调用api：
```py
import os
from openai import OpenAI

client = OpenAI(
    api_key="sk-7d4e4a280bb242bab7361fb9d59c91f7",
    base_url="https://dashscope.aliyuncs.com/compatible-mode/v1",
)
completion = client.chat.completions.create(
		model="qwen3-vl-flash",
    messages=[
        {
            "role": "user",
            "content": [
                {
                	  "type": "image_url", "image_url": {"url": "https://dashscope.oss-cn-beijing.aliyuncs.com/images/dog_and_girl.jpeg"}
                },
                {
                 	 	"type": "text", "text": "这是什么"
                },
            ]
        }
    ]
)
```

**实现**:
- Prompt生成: [`core/prompt_manager.py`](customs_ocr/core/prompt_manager.py:1) 中的 [`generate_prompt()`](customs_ocr/core/prompt_manager.py:13)
- 异步并发识别: [`core/ocr_service.py`](customs_ocr/core/ocr_service.py:1) 中的 [`recognize_images_batch()`](customs_ocr/core/ocr_service.py:18)
- 单图识别: [`recognize_image_async()`](customs_ocr/core/ocr_service.py:47)

**优化**: 使用异步并发，所有图片同时发送给大模型，大幅提升速度

### JSON校验

检验模型输出的JSON是否符合要求，如果不符合，则尝试修复（例如输出可能以```json开头，以```结尾，那么就去掉这些内容），如果修复失败，则重新生成（最多尝试3次），如果3次均失败，则中断，不再执行后续操作

**实现**: [`core/json_utils.py`](customs_ocr/core/json_utils.py:1)
- [`parse_and_validate()`](customs_ocr/core/json_utils.py:16): 解析和验证JSON
- [`remove_markdown_markers()`](customs_ocr/core/json_utils.py:48): 移除markdown标记
- [`extract_json()`](customs_ocr/core/json_utils.py:66): 提取JSON部分
- [`validate_structure()`](customs_ocr/core/json_utils.py:85): 验证JSON结构

### 字段映射与来源验证 ⭐ (新增)

在OCR识别阶段立即进行：
1. **字段名称模糊匹配**: 将大模型输出的keyDesc映射到标准的英文key
   - 精确匹配
   - 去空格匹配
   - 部分匹配（如"毛重（千克）" → "毛重" → grossWt）
2. **来源有效性验证**: 根据Excel定义验证该字段是否应该从当前文件类型识别
   - 例如：净重(netWt)只能从预录入单(4)和装箱单(3)的表头识别
   - 如果在发票中识别到净重，会被过滤掉
3. **过滤无效字段**: 删除key为空或来源无效的字段

**实现**: 
- [`config/field_mapping.py`](customs_ocr/config/field_mapping.py:1)
  - [`fuzzy_match_key_desc()`](customs_ocr/config/field_mapping.py:295): 模糊匹配
  - [`is_valid_source()`](customs_ocr/config/field_mapping.py:280): 来源验证
- [`core/ocr_service.py`](customs_ocr/core/ocr_service.py:93) 中的 [`convert_to_extraction_result()`](customs_ocr/core/ocr_service.py:93)

### 字段内容映射（预留该模块，暂不实现）

调用数据库，将字段映射为标准格式

### 多图片识别内容聚合

1. 对于preDecHead内的字段，按照keyDesc进行聚合，并且根据之前保留的对应关系补充key。同时，将不同来源的识别内容统一记录到sourceList中，并明确不同value和pixel对应的编号（imageId），格式示范如下：
```json
{
  "preDecHead": [
    {
      "keyDesc": "xx",
      "key": "zz",
      "sourceList": [
        {
            "value": "yy1",
            "pixel": [a1, b1, c1, d1],
            "imageId": 1
        },
        {
            "value": "yy2",
            "pixel": [a2, b2, c2, d2],
            "imageId": 2
        }
      ]
    }
  ],
  "preDecList": [
    [
      {
        "keyDesc": "xx",
        "key": "zz",
        "sourceList": [
          {
              "value": "yy1",
              "pixel": [a1, b1, c1, d1],
              "imageId": 1
          },
          {
              "value": "yy2",
              "pixel": [a2, b2, c2, d2],
              "imageId": 2
          }
        ]
      }
    ]
  ]
}
```

2. 对于preDecList内的字段，按商品的识别顺序进行聚合，如果不同图片识别得到的商品数量不一样，则排在后面的商品的聚合来源会更少

**实现**: [`core/aggregator.py`](customs_ocr/core/aggregator.py:1)
- [`aggregate_results()`](customs_ocr/core/aggregator.py:18): 主聚合函数
- [`aggregate_head_fields()`](customs_ocr/core/aggregator.py:39): 表头聚合
- [`aggregate_list_fields()`](customs_ocr/core/aggregator.py:75): 表体聚合

**优化**: key已在OCR阶段映射完成，聚合时直接补充

### 聚合内容检测与转换

1. 检测各个sourceList中的各value是否一致。如果一致，不进行额外处理；如果不一致，利用大模型判断（限定返回结果，如果不符合要求则重试，最多3次）不同value是否代表相同含义（例如考虑是否使用了不同语种、使用了完整名称与缩写等情况），如果是同一个意思，那么就需要修改json文件，把value都设定为同一值

2. 针对修改后的json文件，需要对每个keyDesc额外添加parsedValue。如果sourceList中的value都一样，那么parsedValue就等同于value，如果不一样，则按照规则来选取某个value赋值给parsedValue（预留模块，先直接用sourceList中出现的首个value）

3. 将pixel的[a1,b1,c1,d1]拆分为"startx"、"starty"、"endx"、"endy"，并且根据原有图片的尺寸进行还原（因为大模型的输出坐标值总是0-999，因此需要转换，从而得到原有图片中的坐标框位置），最终输出模板如下：

```json
{
  "preDecHead": [
    {
      "keyDesc": "xx",
      "key": "zz",
      "parsedValue": "vv",
      "sourceList": [
        {
            "value": "yy1",
          	"startx": a1,
          	"starty": b1,
            "endx": c1,
            "endy": d1,
            "imageId": 1
        },
        {
            "value": "yy2",
          	"startx": a2,
          	"starty": b2,
            "endx": c2,
            "endy": d2,
            "imageId": 2
        }
      ]
    }
  ],
  "preDecList": [
    [
      {
        "keyDesc": "xx",
        "key": "zz",
        "parsedValue": "vv",
        "sourceList": [
          {
              "value": "yy1",
              "startx": a1,
              "starty": b1,
              "endx": c1,
              "endy": d1,
              "imageId": 1
          },
          {
              "value": "yy2",
              "startx": a2,
              "starty": b2,
              "endx": c2,
              "endy": d2,
              "imageId": 2
          }
        ]
      }
    ]
  ]
}
```

**实现**:

#### 1. 一致性检测 (异步并发 ⭐)
[`core/aggregator.py`](customs_ocr/core/aggregator.py:1):
- [`check_consistency_and_unify_async()`](customs_ocr/core/aggregator.py:137): 异步批量检查
- [`unify_source_list_async()`](customs_ocr/core/aggregator.py:177): 异步统一单个字段值

#### 2. 智能值比较 ⭐
[`call_llm_to_judge_consistency_async()`](customs_ocr/core/aggregator.py:222):
- **纯数值字段**: 转换为float比较，避免大模型调用
  - `"789.90"` vs `"789.9"` → 转为 789.9 vs 789.9 → 相同，统一为 `"789.90"`
  - `"1,234"` vs `"1234"` → 转为 1234.0 vs 1234.0 → 相同，统一为 `"1,234"`
- **非数值字段**: 使用 qwen-flash 模型异步判断
  - `"USA"` vs `"美国"` → 调用大模型判断
  - `"montreal"` vs `"蒙特利尔"` → 调用大模型判断

#### 3. 后处理
[`core/post_processor.py`](customs_ocr/core/post_processor.py:1):
- [`process_final_output()`](customs_ocr/core/post_processor.py:13): 生成parsedValue和转换坐标
- [`normalize_to_real()`](customs_ocr/core/post_processor.py:113): 坐标转换函数

**坐标转换公式**:
```python
实际坐标 = 归一化坐标 × 图片实际尺寸 / 999
```

## 技术架构

### 双模型策略
- **qwen3-vl-flash**: 图片字段识别（视觉能力强）
- **qwen-flash**: 文本语义判断（速度快、成本低）

### 异步并发架构 ⭐

#### 图片识别阶段
```python
# 所有图片并发识别
tasks = [recognize_image_async(img, prompt) for img, prompt in zip(images, prompts)]
results = await asyncio.gather(*tasks)
```

**性能提升**: N张图片从 N×响应时间 → 1×响应时间

#### 值一致性判断阶段
```python
# 所有字段并发判断
tasks = [unify_source_list_async(field) for field in all_fields]
await asyncio.gather(*tasks)
```

**性能提升**: M个字段从 M×响应时间 → 1×响应时间

**总体提速**: 10-50倍（取决于图片和字段数量）

### 字段验证机制 ⭐

#### 1. 模糊匹配（三级策略）
[`fuzzy_match_key_desc()`](customs_ocr/config/field_mapping.py:295):
1. **精确匹配**: `"净重"` → `netWt`
2. **去空格匹配**: `"净 重"` → `netWt`
3. **部分匹配**: `"净重（千克）"` → `netWt`

#### 2. 来源验证
[`is_valid_source()`](customs_ocr/config/field_mapping.py:280):
```python
# 净重(netWt)只能从预录入单(4)和装箱单(3)的表头识别
is_valid_source("netWt", att_type_code=2, field_type="head") → False  # 发票不能识别净重表头
is_valid_source("netWt", att_type_code=4, field_type="head") → True   # 预录入单可以
```

#### 3. 提前过滤
在 [`convert_to_extraction_result()`](customs_ocr/core/ocr_service.py:93) 中：
- 模糊匹配失败 → 过滤
- 来源验证失败 → 过滤
- 只保留有效字段进入聚合

### 容错机制

1. **JSON解析重试**: 最多3次
2. **自动去除markdown标记**: ```json ... ```
3. **异常隔离**: 单图或单字段失败不影响其他
4. **详细日志**: INFO/WARNING/ERROR/DEBUG四级日志

## 执行流程

```
1. 加载输入数据 (同步)
   └─> load_input_data()

2. 并发识别图片 (异步) ⭐
   ├─> 生成prompts
   └─> recognize_images_batch()
       ├─> recognize_image_async() [图片1]
       ├─> recognize_image_async() [图片2]  } 并发执行
       ├─> recognize_image_async() [图片3]
       └─> ...
           └─> 字段映射+来源验证+过滤

3. 聚合结果 (同步)
   ├─> aggregate_head_fields()  # 按keyDesc分组
   └─> aggregate_list_fields()  # 按商品顺序聚合

4. 并发检查一致性 (异步) ⭐
   └─> check_consistency_and_unify_async()
       ├─> unify_source_list_async() [字段1]
       ├─> unify_source_list_async() [字段2]  } 并发执行
       └─> ...
           ├─> 数值字段：直接比较float
           └─> 非数值字段：call_llm_to_judge_consistency_async()

5. 后处理 (同步)
   ├─> 生成parsedValue
   └─> 坐标转换 [0-999] → 实际像素

6. 输出JSON文件
```

## 数据字典

### 支持的字段类型

#### 表头字段 (preDecHead)
- 运输相关: 运输方式、主运单号、提运单号等
- 贸易相关: 贸易国、运抵国、成交方式等
- 费用相关: 保费、运费、杂费（币制、标记、费率）
- 主体相关: 境内/外收发货人、生产销售单位等
- 监管相关: 监管方式、征免性质、备案号等
- 其他: 件数、毛重、净重、包装种类等

#### 表体字段 (preDecList)
- 商品信息: 商品名称、商品编号、规格型号
- 数量信息: 成交数量、法定数量、件数
- 重量信息: 毛重、净重
- 价格信息: 单价、总价、币制
- 地区信息: 原产国、最终目的国、境内货源地
- 其他: 柜号、征免、成交单位等

### 扩展字段 (extend)
- 合同商品总价 (contrAmount)
- 发票商品总价 (inAmount)
- 装箱单商品净重 (plNetWt)
